# DocCanvas 活体产品工厂剖面设计规格

## 1. 目标

把 DocCanvas 从“缩小的房屋轮廓加卡片网格”重构为一个可读、可操作、有明确生产关系的“活体产品工厂剖面”。产品首先服务 AI 产品负责人和创业者，保留 Markdown 单一事实源、Stage / Module / Track 信息结构以及“整栋产品工厂”隐喻。

本轮设计直接面向当前 Next.js 项目的生产级实施，并沉淀可复用的设计令牌与组件。视觉方向采用可信、克制、温暖的专业产品工厂，同时大胆强化建筑记忆点、2.5D 空间和真人化数字员工。

## 2. 范围与边界

### 2.1 本轮包含

- 工作台 `/`。
- VibeTrack `/canvas/vibe-track`。
- Pro Track `/canvas/v2-pro`。
- Playbook `/canvas/playbook-v2`。
- 桌面端与 390px 移动端布局。
- 新设计令牌、建筑组件、数字员工、2.5D 表达、确定性正交连线、搜索定位、视图保存和导出反馈。
- 真实 Chrome 的逐页视觉与只读功能回归。

### 2.2 本轮不包含

- 不开放生产内容新增、编辑或删除。
- 不修改 Markdown 内容结构、canonical 内容或进化宪章。
- 不启用 owner 写模式，不配置或传输管理 token。
- 不执行部署、容器替换、Nginx 修改、DNS/TLS 修改或生产数据写入。
- 不引入 WebGL、Three.js 或新的 3D 运行时。

边界状态：`production unchanged`。本规格批准不等于实施、部署或生产验收完成。

## 3. 最新浏览器基线证据

2026-07-15 使用用户已打开的 Chrome 对四个页面进行了桌面端与 390×844 移动端只读审计。

### 3.1 已验证事实

- 四个页面均可加载；移动端未发现页面级横向溢出。
- 工作台展示三份内置文档；生产创建表单的标题、说明和创建按钮均为 disabled。
- 节点详情明确显示“只读 / 生产只读”，没有编辑或删除入口。
- Playbook 模块 03 的完整房间含 14 个 React Flow 节点，但 `edgeCount=0`。
- VibeTrack 和 Pro 全景各只有 4 条主连线；路径存在同坐标折返段，没有方向 marker。
- Playbook 完整房间的 H2 为 16px，主要内容卡标题为 12–14px，桌面全景中的建筑标题随画布缩放而变得过小。
- “适应”可以让建筑进入视口，但建筑周围仍有大量无业务意义的留白。
- 搜索“进化宪章”可以返回结果；点击“不可变边界”后只打开所属“安全与治理”房间速读，没有直接打开目标节点详情。
- Markdown 导出实测观察到下载事件。
- 本次 Playbook 速读状态下点击 PNG 后 20 秒内未观察到下载事件，页面也没有错误反馈。该现象需要在本地稳定复现；它不覆盖既有计划文档中的历史 PNG 通过证据。

### 3.2 设计结论

当前主要问题是信息尺度、建筑结构和关系表达，而不是局部颜色：

1. 屋顶为空白梯形，一级标题没有形成品牌入口。
2. 建筑没有有效占据主视口，用户第一眼看不到清晰的全局结构。
3. Playbook 房间没有节点关系线；生命周期画布的主线数量不足且存在退化回折。
4. 搜索结果没有完成“结果 -> 精确节点”的闭环。
5. “保存”容易被理解为内容编辑，但生产实际只读。
6. 页面缺少模块环境和岗位角色，无法自然演进为数字员工产品。

## 4. 已批准的产品决策

| 维度 | 决策 |

| 设计系统起点 | 从零创建新的 DocCanvas 设计系统 |
| 核心用户 | AI 产品负责人 / 创业者 |
| 品牌气质 | 可信、克制、温暖的专业产品工厂 |
| 交付精度 | 生产级高保真 |
| 交付载体 | 直接改造当前 Next.js 项目并沉淀设计令牌与组件 |
| 方案探索 | 三套结构差异明显的方向；最终选择“活体产品工厂剖面” |
| 变化维度 | 布局、视觉、交互、文案、动效、数字员工和创新程度 |
| 必须保留 | Markdown 事实源、Stage / Module / Track、产品工厂隐喻 |
| 创新强度 | 大胆重构，形成强记忆点 |
| 数字员工 | 真人化员工形象 |
| 生产 CRUD | 本轮不实施 |
| 3D 表达 | 轻量 2.5D，不使用可旋转的全 3D 场景 |

## 5. 信息架构

### 5.1 工作台：产品工厂入口大厅

- 一级标题、产品说明和运行边界形成清晰入口。
- 三份 Markdown 文档表现为三座可进入的工厂入口，而不是通用白色列表卡。
- 每个入口显示产品定位、可用状态、更新时间、只读边界和两个主操作：进入、导出 Markdown。
- 生产创建表单本轮不展示。以明确的“生产只读”状态说明代替 disabled 表单，避免制造可写错觉。

### 5.2 桌面总览：建筑剖面

- 标题门楣位于 React Flow 缩放层之外，始终保持可读。
- 屋顶使用低矮工业锯齿轮廓和天窗节奏，不再承担可缩放正文。
- Playbook 使用两层、每层四个能力房间。
- VibeTrack 和 Pro 使用四层、每层两个生命周期房间。
- 建筑中央设置垂直生产主轴；同层使用水平支线。
- 地基固定表达共享能力、安全与治理。
- 入口和附属资源位于首层两侧，不占据主流程房间。
- 默认适配后，建筑主体应占桌面可用画布宽度的 78%–90%、高度的 62%–82%。

### 5.3 房间卡

总览房间只显示：

- Module / Stage 编号。
- 房间名称。
- 数字员工半身像、岗位和一句职责。
- 关键产出摘要。
- Vibe / Shared / Pro 节点统计与资源数量。
- 当前状态：在线、处理中、待验证或受限。

卡片不在总览展示完整节点列表，避免重新形成密集卡片网格。

### 5.4 岗位工作间

点击房间后进入独立岗位工作间，不在总览无限放大：

- 顶部保留房间身份、岗位负责人和返回整栋工厂入口。
- 主区展示当前房间的节点关系。
- 侧区展示选中节点详情、来源章节和资源。
- 搜索结果可以直接进入目标房间并打开目标节点。
- 概览只显示模块级关系；岗位工作间才显示节点级关系。

### 5.5 移动端

- 不缩放桌面 React Flow。
- 使用原生纵向楼层折叠结构。
- 标题、人物、状态和房间统计使用与桌面一致的设计令牌。
- 每个触控目标最小 44×44px。
- 展开楼层后显示房间；进入房间后以全屏工作间展示节点和详情。

## 6. 新设计系统

### 6.1 色彩令牌

设计系统使用暖白建筑基底、石墨文字、森林绿主色和氧化铜强调色：

| 令牌 | 值 | 用途 |
|---|---|---|
| `--factory-canvas` | `#F3F0E7` | 页面与画布背景 |
| `--factory-surface` | `#FCFAF4` | 建筑与面板表面 |
| `--factory-surface-raised` | `#FFFFFF` | 前景内容面 |
| `--factory-ink` | `#1D241F` | 主文字 |
| `--factory-muted` | `#5E6A62` | 次要文字 |
| `--factory-green` | `#28523C` | 品牌、主流程、主操作 |
| `--factory-green-soft` | `#DDE8DE` | 选中背景与状态底色 |
| `--factory-copper` | `#9A583B` | 治理、重点和结构强调 |
| `--factory-slate` | `#53616E` | 依赖关系与工程信息 |
| `--factory-border` | `#C9CDC4` | 建筑边界与分隔 |
| `--factory-danger` | `#9A3F35` | 失败和受限状态 |

实施时必须对文本、按钮、标签和连线进行 WCAG 对比度校验；普通正文至少 4.5:1，大文字至少 3:1。

### 6.2 字体与字号

- 中文主字体栈：`"PingFang SC", "Microsoft YaHei", "Noto Sans CJK SC", sans-serif`。
- 工程标签与编号：`"SFMono-Regular", "Cascadia Code", "Liberation Mono", monospace`。
- 不增加运行时字体依赖；字体加载失败时仍保持稳定排版。
- 桌面 H1：36–44px；移动端 H1：28–32px。
- 桌面 H2：24–30px；房间标题：17–20px；正文：14–16px。
- 工程标签最小 11px，但不承载关键业务信息。

### 6.3 形状与空间

- 建筑层级优先使用梁柱、门框、楼板和分隔线，不使用大量独立圆角白卡。
- 建筑外壳圆角不超过 8px；房间内部面板圆角不超过 6px。
- 阴影分为三层：墙面 0、前景 8px、浮层 16px 的视觉深度。
- 彩色左侧竖条不再作为默认模块识别方式。
- 装饰不得遮挡标题、状态、统计或连线。

## 7. 2.5D 空间规范

- 使用 CSS 分层、墙体厚度、投影、遮挡和轻微视差形成 2.5D。
- 页面保持正视阅读；不允许用户任意旋转相机或倾斜文字。
- 建筑容器使用统一透视，房间正文保持二维平面。
- 鼠标视差最大 6px；移动端禁用视差。
- 进入房间的镜头变化控制在 240ms 内，只使用平移、缩放和景深变化。
- `prefers-reduced-motion: reduce` 时取消视差和镜头缩放，只保留即时状态切换。
- 低性能或图片加载失败时，建筑仍以静态二维层级完整可用。

## 8. 数字员工体系

### 8.1 角色资产

- 建立八名共享数字员工，对应产品工厂的八类核心岗位。
- 三份文档按语义复用角色，不为每份文档复制一套人物。
- 人物为真人化合成形象，不使用真实员工身份或未经授权的照片。
- 统一镜头高度、柔和侧光、服装体系、背景色温和半身构图，避免图库拼贴感。
- 原始资产使用 4:5 构图；交付 WebP/AVIF 响应式尺寸，单张首屏候选资源目标不超过 140KB。

### 8.2 岗位模型

初始岗位库：

1. 产品导航顾问。
2. 工厂运营设计师。
3. 产品架构师 / 知识工程师。
4. 安全治理官。
5. 进化评估师。
6. 交付工程师。
7. 经营分析师。
8. 边界审计官。

每名角色包含稳定 ID、姓名或岗位称谓、岗位、职责、状态、portrait key 和环境 key。缺失人物资源时显示正式岗位占位卡，并在开发日志中记录缺失 key。

### 8.3 房间环境

每个模块具有低对比度环境背景，例如档案室、安全控制室、交付车间或经营分析室。背景只表达语义，不承载正文；文字区域必须有稳定表面层。

## 9. 连线规范

### 9.1 关系类型

| 类型 | 样式 | 默认可见性 |
|---|---|---|
| 主流程 | 3px 森林绿实线、方向箭头 | 总览可见 |
| 依赖 | 2px 石墨灰虚线、方向箭头 | 总览可见 |
| 治理约束 | 2px 氧化铜点划线、方向箭头 | 总览可见 |
| 资源引用 | 1.5px 中性线 | 选中后显示 |

### 9.2 路由约束

- 只使用上、右、下、左四向端口。
- 只生成水平和垂直正交段。
- 不生成长度为零的段、原地折返或同一坐标的反向段。
- 连线不得穿过房间正文和人物面部。
- 跨楼层关系优先汇入垂直主轴。
- 同向关系优先合并为公共管线，再分流到目标。
- 路由使用稳定排序和确定性偏移，不使用 `Math.random()`。
- 方向箭头与线型共同表达语义，不能只靠颜色。
- 悬停房间时高亮完整上下游路径；点击后锁定并淡化无关关系。

### 9.3 语义缩放

- 整栋总览：模块级关系。
- 岗位工作间：节点级主关系。
- 选中节点：资源引用和完整上下游。

## 10. 交互设计

### 10.1 全局工具栏

- 工作台。
- 搜索。
- 适应建筑。
- 重置视图。
- 保存视图。
- 导出 PNG。
- 导出 Markdown。

“保存”必须重命名为“保存视图”，仅保存缩放、位置、楼层展开和当前房间，不暗示修改 Markdown。

### 10.2 搜索闭环

1. 输入标题或正文关键词。
2. 结果展示节点标题、所属房间、来源章节和摘要。
3. 点击结果后关闭搜索。
4. 自动进入所属岗位工作间。
5. 打开并高亮精确目标节点。
6. 返回时保留原搜索词和结果上下文。

### 10.3 动效

- Hover：160ms。
- 选择、高亮和楼层展开：180–220ms。
- 进入岗位工作间：220–240ms。
- 不使用循环漂浮、脉冲或闪烁。
- 状态变化必须有文本或图形反馈，不能只依赖动画。

### 10.4 导出反馈

- 点击导出后立即显示“正在生成”。
- 成功后显示文件类型和完成状态。
- 失败后显示可读错误与重试入口。
- PNG 生成失败不得静默；控制台错误也不能替代用户反馈。

## 11. 技术架构

### 11.1 保留的主链路

```text
Markdown
  -> extractMarkdownSections() / parseMarkdownToGraph()
  -> buildArchitectureViewModel()
  -> computeArchitectureLayout()
  -> Factory presentation projection
  -> React Flow desktop / native mobile navigation
```

Markdown 继续是内容事实源；展示元数据不回写 Markdown。

### 11.2 新的展示边界

建议新增以下独立单元，具体文件名在实施计划中与现有代码再次核对：

- Factory design tokens：颜色、字体、空间、深度、动效和连线。
- Factory presentation registry：语义 ID 到岗位、人物、环境和关系类型的确定性映射。
- Factory shell：固定门楣、工具栏和画布边界。
- Factory building：屋顶、楼层、房间、地基和入口。
- Factory pipeline layer：模块级与节点级正交路由。
- Room workspace：岗位工作间和节点详情。
- Digital employee：人物、岗位、状态和资源回退。
- Mobile factory navigator：不依赖 React Flow 的移动端楼层导航。
- Export feedback：PNG / Markdown 的统一状态与错误反馈。

每个单元只承担一种职责；桌面和移动端共享同一 ViewModel 与展示元数据，不各自重新分类资源或角色。

### 11.3 展示元数据契约

展示注册表以现有稳定语义 ID 为 key，至少包含：

```ts
type FactoryPresentation = {
  semanticId: string;
  employeeRoleId: string;
  environmentKey: string;
  accentToken: string;
};

type FactoryRelation = {
  id: string;
  sourceSemanticId: string;
  targetSemanticId: string;
  kind: "flow" | "dependency" | "governance" | "resource";
};
```

注册表不得复制文档路径映射；文档身份继续来自 `lib/shared/document-registry.ts`。

## 12. 错误处理与降级

- 人物资源缺失：显示岗位占位卡，记录缺失 key，页面继续可读。
- 环境背景缺失：显示稳定的建筑表面色，不显示破图。
- 搜索无结果：显示搜索范围和清除入口。
- 精确目标不存在：保留搜索结果并提示内容可能已变化，不猜测其他 heading。
- 视图保存失败：使用既有 localStorage 兜底与重试契约，并明确提示当前仅保存到本机。
- PNG / Markdown 导出失败：显示失败原因和重试，不静默吞错。
- reduced motion 或低性能：关闭视差和镜头变化，保留全部内容与操作。

## 13. 验收标准

### 13.1 自动化

- `npm run typecheck`。
- 相关 node:test 回归。
- 核心用户可见行为变化完成后运行 `npm run verify:local`。
- 项目没有 lint script，不声明 lint 通过。

新增或强化以下测试：

1. 展示注册表覆盖所有内置 Module / Stage，且无重复 key。
2. 数字员工和环境映射为确定性输出。
3. 正交路由没有对角线、零长度段和原地折返。
4. 公共管线合并后目标仍保持正确方向。
5. 搜索结果直接进入目标节点详情。
6. 保存视图不调用 Markdown 写 API。
7. 生产模式不展示内容 CRUD。
8. Markdown 与 PNG 均触发下载并显示成功状态。
9. PNG 失败路径显示用户可见错误。
10. reduced motion 关闭视差与镜头动画。

### 13.2 浏览器视觉回归

使用真实 Chrome 验证：

- 工作台、VibeTrack、Pro、Playbook。
- 默认桌面视口和 390×844 移动视口。
- 无页面级横向溢出。
- 一级标题在桌面不小于 36px，在移动端不小于 28px。
- 默认适应后建筑主体满足既定视口占比。
- 连接线具有方向，不穿过正文，无退化折返。
- 房间人物不遮挡标题、状态或统计。
- 搜索精确定位、视图保存、Markdown 和 PNG 导出可用。
- 控制台无新增 error；warning 必须分类并处理。
- 键盘可达，焦点清晰，移动触控目标至少 44px。

### 13.3 证据分层

- 本地 typecheck/test/build 只证明本地实现层。
- 本地 Chrome 视觉回归只证明本地运行层。
- 候选包 smoke 不等于生产上线。
- 生产浏览器验收、部署和 owner 写模式均需单独授权与新鲜证据。

## 14. 实施顺序建议

1. 冻结浏览器基线和设计令牌。
2. 建立展示注册表与八名数字员工资产契约。
3. 重构固定门楣、屋顶和建筑外壳。
4. 重构楼层、房间和地基。
5. 实现确定性生产主轴与正交路由。
6. 实现岗位工作间和精确搜索定位。

详细文件级步骤必须在本规格通过用户复核后另行制定实施计划。

## 15. 完成定义

只有同时满足以下条件，才能声明本轮实现完成：

- 四个页面完成桌面与移动端改造。
- 设计令牌与共享组件已落地，不是页面级重复样式。
- 八名数字员工及其回退状态可用。
- 总览和岗位工作间的连线满足确定性正交规范。
- 搜索精确打开目标节点。
- Markdown 与 PNG 导出均有成功和失败反馈。
- 相关测试、typecheck、build 与真实 Chrome 回归通过。
- 未越过生产只读边界。

部署、生产 activation 和无边界的 `production_accepted` 不属于本规格的完成声明。
